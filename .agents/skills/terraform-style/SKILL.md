# Terraform Style Guide for Power Platform Modules

> Adapted from the HashiCorp `terraform-style-guide` skill for Power Platform community modules.

## File Organization

Every module **must** contain these files:

| File           | Purpose                                                            |
| -------------- | ------------------------------------------------------------------ |
| `terraform.tf` | `terraform` block with `required_version` and `required_providers` |
| `main.tf`      | Primary resource definitions                                       |
| `variables.tf` | All input variable declarations                                    |
| `outputs.tf`   | All output value declarations                                      |
| `locals.tf`    | Local value computations                                           |
| `_header.md`   | terraform-docs header partial                                      |
| `_footer.md`   | terraform-docs footer partial                                      |
| `README.md`    | Auto-generated by terraform-docs (never edit manually)             |

### Additional files (as needed)

- `data.tf` — Data source definitions
- `moved.tf` — State migration blocks (`moved`, `import`)

### File naming

- Use **lowercase** with hyphens for multi-word filenames
- Use `.tf` extension for all Terraform configuration files
- Use `.tftest.hcl` extension for test files
- Use `.tfvars` extension for variable value files

## Naming Conventions

### Resources and Data Sources

```hcl
# Use descriptive names — avoid generic "this" unless the module manages a single resource
resource "powerplatform_environment" "development" {
  # ...
}

# For single-resource modules, "this" is acceptable
resource "powerplatform_tenant_settings" "this" {
  # ...
}
```

### Variables

```hcl
# Use lowercase with underscores
variable "environment_display_name" {
  description = "The display name of the Power Platform environment."
  type        = string
}

# Group related variables with a common prefix
variable "environment_location" { ... }
variable "environment_type" { ... }
```

### Outputs

```hcl
# Mirror the resource attribute name where possible
output "environment_id" {
  description = "The ID of the Power Platform environment."
  value       = powerplatform_environment.this.id
}
```

### Locals

```hcl
# Use descriptive names that explain the computation
locals {
  environment_url = "https://${var.environment_domain}.crm.dynamics.com"
}
```

## Formatting

- Run `terraform fmt -recursive` before every commit
- Use 2-space indentation (enforced by `.editorconfig`)
- Align `=` signs within blocks for readability
- Place blank lines between top-level blocks
- Order arguments within resource blocks (AVM TFNFR8):
  1. Meta-arguments (top): `provider`, `count`, `for_each`
  2. Required arguments (alphabetical)
  3. Optional arguments (alphabetical)
  4. Required nested blocks (alphabetical)
  5. Optional nested blocks (alphabetical)
  6. Meta-arguments (bottom): `depends_on`, `lifecycle` (sub-order: `create_before_destroy`, `ignore_changes`, `prevent_destroy`)

## Variable Requirements

Every variable declaration **must** include:

```hcl
variable "name" {
  description = "Clear, concise description of what this variable controls."
  type        = string  # Always specify type

  # Validation block for constrained inputs
  validation {
    condition     = length(var.name) > 0
    error_message = "Name must not be empty."
  }
}
```

### Type hierarchy preference

1. `string`, `number`, `bool` — Simple types
2. `list(string)`, `map(string)` — Simple collections
3. `object({...})` — Structured types (use for complex inputs)
4. `any` — Avoid unless absolutely necessary

## Output Requirements

Every output declaration **must** include:

```hcl
output "resource_id" {
  description = "The unique identifier of the resource."
  value       = powerplatform_environment.this.id
}
```

## Version Pinning

### Terraform version

```hcl
terraform {
  required_version = ">= 1.9, < 2.0"
}
```

### Provider versions

```hcl
terraform {
  required_providers {
    powerplatform = {
      source  = "microsoft/power-platform"
      version = "~> 4.0"
    }
  }
}
```

- Use **pessimistic constraint** (`~>`) for providers
- Never use `>=` without an upper bound for providers

## Security Best Practices

1. **Never hardcode credentials** in `.tf` files
2. **Mark sensitive variables** with `sensitive = true`
3. **Mark sensitive outputs** with `sensitive = true`
4. **Use OIDC authentication** for CI/CD pipelines — no client secrets
5. **Pin all versions** — Terraform, providers, and modules
6. **Review provider permissions** — Use least-privilege service principals
7. **Validate inputs** — Use `validation` blocks to reject invalid values early
8. **Exclude secrets from state** — Consider `lifecycle { ignore_changes }` for credential fields

## Power Platform Provider Specifics

- Provider source: `microsoft/power-platform`
- Authentication: OIDC (preferred), client credentials, or interactive
- Resource naming: `powerplatform_*` prefix
- Common resources: `powerplatform_environment`, `powerplatform_tenant_settings`, `powerplatform_data_loss_prevention_policy`
- Environment IDs are UUIDs — always validate format
